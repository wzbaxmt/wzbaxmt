使用CPU硬件指令加速AES加密

1 引言

AES是世界上最安全、使用广泛的加密算法，很多安全合规要求里面都明确要求使用AES算法，只是相对于3des、rc4等加密算法，速度慢了很多，幸好有了AES-NI，这是针对AES加密算法的硬件加解密CPU指令集。

AES-NI的全称是：Advanced Encryption Standard New Instructions。

支持AES-NI指令集的 Intel CPU列表 ，基本上2010年之后的Intel CPU都支持，另外AMD和ARM的一些型号也支持

本文将使用openssl自带的评测加密速度功能进行CPU硬件加速前后加密速度的比较，使用的是AES_CBC算法。

2测试环境与配置

服务器配置：

型号：浪潮英信NP3020M4

CPU：Intel(R) Xeon(R) CPU E3-1220 v5 @3.00GHz

内存：8GB

硬盘：1TB

系统：CentOS release 6.9(Final)

应用软件：无（除装机自带软件）

网络：无

3测试方法

测试单核性能：

  1. 进入CentOS系统，打开终端。
  2. 输入命令lscpu查看CPU信息。
  3. 输入命令openssl speed -elapsed -evp aes-256-cbc，测试在开启AES-NI指令加速情况下的AES256位的加密速度。
  4. 输入命令OPENSSL_ia32cap="~0x200000200000000" openssl speed -elapsed -evp aes-256-cbc，测试在关闭AES-NI指令加速情况下的AES256位的加密速度。
  5. 输入命令openssl speed -elapsed -evp aes-128-cbc，测试在开启AES-NI指令加速情况下的AES128位的加密速度。
  6. 输入命令OPENSSL_ia32cap="~0x200000200000000" openssl speed -elapsed -evp aes-128-cbc，测试在关闭AES-NI指令加速情况下的AES128位的加密速度。
  7. 保存并记录。

  1. 密钥位数越多，加密字符串长度越长，AES加密处理速度越慢。
  2. 开启CPU的AES加速（AES-NI）功能后，AES加密速度约提升为原来的3.6倍。

  4. 进程数超过CPU核心数后加密速度不再提升。

7 源代码及其他

openssl-1.0.2l\apps\speed.c

由代码可知，未加互斥锁等线程同步方法，判断该测试程序为单线程程序，故测试值的含义是单个CPU处理的数据量。

在CBC模式下，每个明文块在加密之前与先前的密文块进行异或。这样，每个密文块依赖于直到那一点处理的所有明文块，为了使每个消息唯一，必须在第一个块中使用初始化向量，CBC一直是最常用的操作模式，其主要缺点是加密是连续的（即它不能并行化），并且消息必须被填充到密码块大小的倍数。（源自《Block cipher mode of operation》，Cipher Block Chaining (CBC)）

因此对单独文件的加密不能采用多线程分块加密后合并的方法， 只能进行单线程串行处理。 同时对多个文件加密时可采用多进程的方法。

使用openssl的AES-CBC算法进行加密时，单个加密进程会占用单独的CPU（可在加密时用 mpstat –P ALL 1 命令查看CPU各核心占用 ），而不会像windows下各CPU平均分配程序运行时间，该算法在Centos6.9系统下的服务器上运行时为单核运算。

然而，在本机硬件系统下（机械硬盘），写入速度173MB/s，加密速度远大于硬盘写入速度，即使是固态硬盘，目前其写入速度基本为400MB/s，仍小于单核加密速度，因此加密性能瓶颈其实在硬盘端。

实际加密测试也验证了该观点，进行128位AES加密时CPU占用在18%左右，约等于硬盘写入速度，而即使开启多个加密进程，因为硬盘写入速度的限制，其CPU使用率也不会有提升，多个加密进程总和仍为18%左右，而通过查看硬盘写入速度（iostat –d –k 1命令查看实时硬盘数据吞吐量）可以验证这个结论，无论加密进程数的多少，硬盘写入数据量始终维持在75M/s左右。
